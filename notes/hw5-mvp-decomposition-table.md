# Декомпозиция MVP — CampusBook

## 1. Продукт и сценарий

**Главная задача MVP:** Дать студентам возможность увидеть свободные ресурсы кампуса и забронировать их за 30 секунд — без переписок в чатах и конфликтов.

**Пользователь:** Студент университета (18–25 лет), которому нужна переговорка/коворкинг/оборудование для командной работы.

**Ключевой сценарий для защиты:**
Студент открывает CampusBook → видит каталог ресурсов → фильтрует по типу → выбирает переговорную → указывает дату и время → бронирует → видит бронь в разделе «Мои бронирования» → при необходимости отменяет.

---

## 2. Пользователи

| Роль | Описание | Возможности |
|------|----------|-------------|
| **Гость** | Любой, кто открыл сайт без авторизации | Просмотр каталога ресурсов, фильтрация, поиск. Не может бронировать |
| **Студент** (основная роль) | Авторизованный пользователь | Всё что гость + бронирование, просмотр своих броней, отмена, экспорт |
| **Преподаватель** | Авторизованный пользователь с расширенными правами | Всё что студент + публикация слотов для консультаций |
| **Администратор** | Управляющий системой | Всё + CRUD ресурсов, просмотр всех бронирований, статистика загрузки |

*В текущем MVP: роли гость и студент. Авторизация упрощена — имя вводится в форме бронирования.*

## 8. Хранение данных

| Что хранится | Где | Почему | Сохраняется после перезагрузки? |
|-------------|-----|--------|-------------------------------|
| Список ресурсов | JS-константа в `script.js` | Статические данные, редко меняются | Да (зашиты в коде) |
| Бронирования | `localStorage` (ключ `campusbook_bookings`) | Динамические данные пользователя | Да |
| Текущий фильтр и поисковый запрос | Переменные JS в памяти | Временное состояние интерфейса | Нет (сбрасывается) |
| Состояние формы | DOM (значения полей) | Ввод пользователя | Нет |

**При миграции на Supabase:** ресурсы и бронирования переедут в PostgreSQL, пользователи будут храниться через Supabase Auth.

## 9. Backend / логика

| Сценарий | Что делает система | Где реализовано |
|----------|-------------------|-----------------|
| Создание брони | 1. Читает данные формы → 2. Проверяет startTime < endTime → 3. Загружает все брони из localStorage → 4. Ищет конфликт (тот же ресурс + дата + пересечение времени) → 5. Если ОК — сохраняет новую бронь, если нет — показывает ошибку | `handleFormSubmit()` |
| Отмена брони | 1. Находит бронь по ID → 2. Меняет статус на `cancelled` → 3. Сохраняет в localStorage → 4. Перерисовывает список и статистику | `cancelBooking()` |
| Фильтрация каталога | 1. Получает тип фильтра → 2. Фильтрует массив RESOURCES → 3. Применяет текстовый поиск поверх → 4. Перерисовывает карточки | `renderCards()` |
| Подсчёт статистики | 1. Загружает все брони → 2. Считает total / active / cancelled → 3. Рендерит счётчики | `renderStats()` |
| Экспорт данных | 1. Читает localStorage → 2. Создаёт Blob с JSON → 3. Инициирует скачивание файла | `exportBookings()` |

## 10. API (черновик)

*В текущем MVP всё работает на клиенте. При переходе на Supabase потребуются следующие запросы:*

| Действие | Метод | Endpoint | Тело запроса | Ответ |
|----------|-------|----------|-------------|-------|
| Получить все ресурсы | GET | `/resources` | — | `[{id, name, type, ...}]` |
| Получить брони за дату | GET | `/bookings?resourceId=...&date=...` | — | `[{id, startTime, endTime, ...}]` |
| Создать бронь | POST | `/bookings` | `{resourceId, date, startTime, endTime, purpose}` | `{id, status: "active"}` или `{error: "conflict"}` |
| Отменить бронь | PATCH | `/bookings/:id` | `{status: "cancelled"}` | `{ok: true}` |
| Получить мои брони | GET | `/bookings?userId=...` | — | `[{id, resourceId, date, ...}]` |

## 11. Проверки и правила

| Проверка | Когда | Что происходит при нарушении |
|----------|-------|------------------------------|
| Все поля формы заполнены | При отправке формы | HTML5 `required` не даёт отправить форму |
| Время начала < время конца | При отправке формы | Красное уведомление: «Время начала должно быть раньше времени окончания» |
| Нет пересечения с существующей бронью | При отправке формы | Красное уведомление с деталями конфликта |
| Дата не в прошлом | При выборе даты | Атрибут `min` на `<input type="date">` — прошлые даты недоступны |
| Отменить можно только активную бронь | При нажатии «Отменить» | Кнопка не рендерится для отменённых бронирований |

## 12. Авторизация

| Состояние | Что может делать | Как реализовано в MVP |
|-----------|-----------------|----------------------|
| **Без авторизации** (текущий MVP) | Всё — каталог, бронирование, управление | Имя вводится вручную в форме. Все данные хранятся локально в браузере |
| **Гость** (план) | Просмотр каталога, поиск, фильтрация | Доступ без логина |
| **Авторизованный пользователь** (план) | Бронирование, личный кабинет, отмена | Supabase Auth (email + пароль) |
| **Админ** (план) | CRUD ресурсов, просмотр всех броней, статистика | Роль в Supabase, отдельная админ-страница |

**Auth flow (план):**
Открыл сайт → видит каталог как гость → нажал «Забронировать» → перенаправлен на страницу входа → ввёл email/пароль → вернулся к форме → бронирует.

## 13. Пользовательский поток

```
Открыл CampusBook
       │
       ▼
Видит Hero-секцию → нажимает CTA «Забронировать»
       │
       ▼
Скроллит к каталогу → фильтрует/ищет нужный ресурс
       │
       ▼
Скроллит к форме бронирования
       │
       ▼
Заполняет: имя, ресурс, дата, время, цель
       │
       ▼
Нажимает «Забронировать»
       │
       ├── Успех → зелёное уведомление, бронь в списке
       │
       └── Ошибка → красное уведомление
              │
              ├── Конфликт → «Ресурс занят в это время»
              └── Невалидное время → «Начало позже конца»
                     │
                     ▼
              Исправляет → повторяет
       │
       ▼
Скроллит к «Мои бронирования» → видит бронь
       │
       ├── Всё ОК → приходит в забронированное место
       └── Планы изменились → нажимает «Отменить»
```

**Где может «сломаться»:**
- Пользователь не заполнил обязательное поле → HTML-валидация не даёт отправить
- Пользователь выбрал конец раньше начала → JS-валидация показывает ошибку
- Ресурс уже занят → проверка конфликтов предотвращает двойное бронирование
- localStorage переполнен → маловероятно при нормальном использовании (лимит ~5 МБ)

## 14. Ошибки и крайние случаи

| Ситуация | Что произойдёт | Как обработано |
|----------|----------------|----------------|
| Поле формы не заполнено | Форма не отправляется | HTML5 `required` |
| Время начала >= время конца | Показывается ошибка | JS-валидация в `handleFormSubmit()` |
| Ресурс уже забронирован на это время | Показывается конфликт с деталями | JS-проверка пересечения слотов |
| localStorage пуст (первый визит) | Пустой список, сообщение «Пока нет бронирований» | Fallback в `loadBookings()` → возвращает `[]` |
| localStorage повреждён (невалидный JSON) | Данные сбрасываются | `try/catch` в `loadBookings()` |
| Нет интернета | Сайт работает полностью (если уже загружен) | Всё на клиенте, единственная зависимость — фото с Unsplash (не критично) |
| Изображение ресурса не загрузилось | Показывается серый плейсхолдер | CSS: `background: #e2e8f0` на `.card-img` |
| Пользователь пытается бронировать прошлую дату | Прошлые даты недоступны | `min` атрибут на `<input type="date">` |

## 15. MVP-границы

### Обязательно работает (без этого нельзя показывать)

- Каталог ресурсов отображается с фото и описанием
- Фильтрация по типу и поиск по тексту работают
- Форма бронирования сохраняет данные в localStorage
- Защита от конфликтов (нельзя забронировать занятый слот)
- Список бронирований отображается, можно отменить
- Данные сохраняются после перезагрузки страницы

### Можно отложить (backlog)

- Авторизация (email/пароль)
- Серверная база данных (Supabase)
- Админ-панель (CRUD ресурсов)
- Уведомления (email, push, Telegram)
- Повторяющиеся бронирования
- Мобильное приложение
- Интеграция с университетской SSO

## 16. Публикация

| Вариант | Подходит? | Как это работает |
|---------|-----------|------------------|
| **Локальный запуск** | Да (текущий MVP) | Открыть `index.html` в браузере двойным кликом. Работает без сервера |
| **GitHub Pages** | Да | Включить Pages в настройках репо → сайт доступен по `https://vv0rkz.github.io/vibe-coding-campus-booking-for-innopolis-course/projects/campus-booking/` |
| **Replit** | Да | Import from GitHub → запускается автоматически, даёт публичную ссылку |
| **Vercel / Netlify** | Да (для следующего этапа) | Подключить репо → автодеплой при каждом push |

**Для защиты:** GitHub Pages — самый простой вариант. Преподаватель открывает ссылку → видит рабочий лендинг → может забронировать ресурс и проверить localStorage.
