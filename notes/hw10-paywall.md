# Домашнее задание 10 — Paywall и первая монетизация

## Проект: CampusBook — сервис бронирования ресурсов кампуса

---

## Идея

Переход от «пользователь» → «клиент». Бесплатный доступ с лимитом, дальше — экран оплаты и разблокировка.

**Формула:** сначала ценность, потом деньги.

Пользователь сначала спокойно бронирует ресурсы, понимает пользу продукта, и только после нескольких броней упирается в лимит Free-тарифа — тогда мы предлагаем PRO.

---

## Продуктовая логика

| Тариф | Лимит | Фичи |
|---|---|---|
| **Free** | до 3 активных броней | Каталог, календарь, бронирования, отмена |
| **PRO** (199 ₽/мес) | без лимита | + приоритет, + интеграция Google Calendar в каждой брони, + будущие фичи |

Логика:
1. Пользователь создаёт бронь → при submit фронт считает сколько у него сейчас **активных** броней.
2. Если активных ≥ 3 **и** `profiles.paid = false` → показываем paywall-модалку.
3. Paywall: «Оплатить 199 ₽» → редирект на `/success.html` (заглушка тестовой оплаты).

---

## Схема БД

```sql
create table public.profiles (
  id uuid primary key references auth.users (id) on delete cascade,
  email text,
  display_name text,
  paid boolean not null default false,
  ...
);
```

Полный DDL + RLS — в [`projects/campus-booking/supabase/schema.sql`](../projects/campus-booking/supabase/schema.sql).

**Важно:** политика `profiles_update_own_name` запрещает фронту менять `paid`. Это принципиально — иначе любой пользователь через DevTools мог бы обновить свою строку и получить PRO. Обновление `paid` должно идти через **service_role** (webhook платёжного провайдера, см. hw11).

## UI-элементы

### 1. Paywall-модалка (`#paywall-modal` в `index.html`)

- Заголовок «Бесплатный доступ закончился».
- Текст с лимитом.
- Буллеты фич PRO.
- Кнопка «Оплатить 199 ₽» → редирект на `success.html`.

Триггеры показа:
- В `handleBookingSubmit`: если `useSupabase && !profile.paid && activeBookings >= FREE_BOOKING_LIMIT`.

### 2. Плашка тарифа в профиле

Модалка «Мой профиль» отображает текущий план:
- «Free — до 3 активных броней» (обычный фон),
- «PRO — полный доступ» (зелёный фон).

Тариф также виден в dropdown-меню профиля (строка `Free` / `PRO` в мини-счётчиках).

### 3. `success.html` (страница «Оплата почти завершена»)

Отдельная HTML-страница. Показывает:
- иконку ожидания,
- текст: «Мы получили запрос на оплату. Доступ активируется после подтверждения платежа.»,
- ссылку «Вернуться в приложение» → `index.html`.

**Что эта страница НЕ делает:** не обновляет `profiles.paid = true`. В учебной версии по лекции здесь должен был быть автоматический UPDATE с фронта, но это ломает безопасность — любой может открыть URL `/success.html` и получить PRO бесплатно (это демонстрируется в hw11 «Взлом paywall»). Поэтому сразу сделали правильно: обновление `paid` планируется webhook-ом на серверной функции Vercel, а `success.html` — просто landing.

## Интеграция платежей (реальная — в плане)

| Провайдер | API | Комиссия | Под что |
|---|---|---|---|
| **ЮKassa** | REST + Webhook | 3.5% | РФ физики/юрики |
| **CloudPayments** | Виджет + Webhook | 3.2% | РФ, подписки |
| **Stripe** | API + Webhook | 2.9% + $0.30 | Не РФ |

Упрощённый flow (для hw11):

```
Кнопка «Оплатить» (клиент)
    │
    ▼
POST /api/create-payment (Vercel Serverless) → ЮKassa API
    │
    ▼
ЮKassa возвращает confirmation_url, клиент редиректится
    │
    ▼
После оплаты клиент → /success.html (просто landing)
    │
    ▼
ЮKassa отправляет webhook POST /api/payment-webhook (Vercel)
    │
    ▼
Webhook проверяет подпись, через service_role Supabase ставит profiles.paid = true
```

## Что реализовано сейчас

| Пункт из лекции | Статус | Где |
|---|---|---|
| Таблица `profiles` | ✅ | `supabase/schema.sql` |
| При входе — создать/загрузить профиль | ✅ | `loadProfile()` в `script.js` |
| Если `paid=false` — показать paywall при лимите | ✅ | `handleBookingSubmit` |
| Если `paid=true` — обычный доступ | ✅ | тот же код |
| Блок paywall с «Оплатить 199 ₽» | ✅ | `#paywall-modal` |
| Кнопка оплаты → `/success` | ✅ | `window.location.href = 'success.html'` |
| Страница `/success` **автоматически** ставит `paid=true` | ❌ **намеренно** | небезопасно, см. hw11 |

Отличие от чистой реализации по слайдам: мы не делаем auto-paid-update, сразу идём к hw11-версии. В hw10 отмечаем это сознательное решение.

## Тестирование

| Сценарий | Ожидание |
|---|---|
| Свежий пользователь, 0 броней | Форма бронирования работает, paywall не показывается |
| Free-пользователь сделал 2 активные брони | Третья проходит |
| Free-пользователь сделал 3 активные брони, пробует четвёртую | Открывается paywall |
| Отменил одну из 3 активных → попробовал создать новую | Проходит (осталось 2 активных) |
| Прямой переход на `/success.html` без оплаты | Видит «оплата почти завершена», `paid` **не** меняется |
| Пользователь с `paid=true` (выставлен вручную через SQL Editor) | Paywall не показывается даже на 10+ бронях |

Ручной апгрейд в тесте: через **Supabase → Table Editor → profiles** поставить `paid = true` на своей строке и обновить страницу.
