# Домашнее задание 7 — Auth flow, роли, приватные разделы, профиль

## Проект: CampusBook — сервис бронирования ресурсов кампуса

---

## Цель

Развить базовую email-авторизацию из hw6 в полноценный auth flow: отдельный профиль пользователя, роли, приватные разделы, более аккуратный UX входа/регистрации.

## Что было после hw6

- `bookings` + RLS, вход по email/паролю, бронирование только для залогиненных.
- В хедере висели две кнопки «Войти» и «Выйти» рядом.
- Регистрация и вход были в одной форме без валидации (только HTML `required`/`minlength`).
- Ни профиля, ни отображаемого имени — везде использовался сырой email.

## Что добавлено в hw7

### 1. Таблица `profiles`

```sql
create table public.profiles (
  id uuid primary key references auth.users (id) on delete cascade,
  email text,
  display_name text,
  paid boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

RLS:
- SELECT — только свой профиль (`id = auth.uid()`).
- INSERT — только свой, и только с `paid = false` (фронт не может сам сделать себя PRO).
- UPDATE — свой, но менять `display_name`, не `paid` (поле `paid` обновляется серверным webhook платёжки, см. hw11).

Полный скрипт: [`projects/campus-booking/supabase/schema.sql`](../projects/campus-booking/supabase/schema.sql).

### 2. Жизненный цикл профиля

При первом входе клиент пытается прочитать свою строку из `profiles`. Если её нет — создаёт с `display_name`, взятым либо из `user_metadata.display_name` (пришёл из формы регистрации), либо из локальной части email до `@`. См. `loadProfile()` в `script.js`.

### 3. Роли

| Роль | Что может | Как реализовано |
|------|-----------|-----------------|
| Гость | Смотрит каталог, календарь (без видимости деталей броней) | Supabase-клиент без сессии, RLS закрывает чужие данные |
| Пользователь | Создаёт брони, видит свои, отменяет, редактирует `display_name`, переключает тариф через paywall | Session + `profiles` + RLS |
| Админ | CRUD ресурсов, все брони (план) | Роль в Supabase, отдельный UI — ещё не сделано |

Роль админа пока не реализована: каталог ресурсов — константа в `script.js`, не таблица, поэтому CRUD-админки физически нет. Следующий шаг — вынести `RESOURCES` в таблицу `resources` и сделать политику «админ читает-пишет всё».

### 4. Приватные разделы

- Секция **«Забронировать»** (`#booking`) — для гостя показывает gate с кнопкой «Войти / Регистрация».
- Секция **«Мои бронирования»** — для гостя показывает «Войдите, чтобы увидеть свои бронирования».
- Модалка **«Мой профиль»** — вообще доступна только из меню авторизованного пользователя (меню скрыто у гостя).
- Paywall-модалка — показывается только когда пользователь упёрся в лимит free-броней (hw10).

### 5. UI auth-flow

Хедер переделан:
- **Гость** — одна компактная синяя кнопка «Войти» с иконкой пользователя.
- **Пользователь** — аватар-кружок с инициалами от имени; клик открывает dropdown-меню с именем, email, мини-статистикой (активные / всего / тариф), пунктом «Мой профиль» и кнопкой «Выйти».

Это убирает визуальный шум: вместо двух кнопок рядом («Войти», «Выйти» — обычно только одна из них нужна) — одна сущность, состояние которой показывает текущий статус.

### 6. Форма регистрации/входа

- Табы «Вход» / «Регистрация» (как раньше), но регистрация теперь раскрывает два дополнительных поля: имя и подтверждение пароля.
- Валидация на клиенте (hw7) — подробности в `hw7-auth-flow.md#валидация`:
  - email: `type="email"` + `pattern="[^\s@]+@[^\s@]+\.[^\s@]+"`,
  - пароль: `minlength=6`, `maxlength=72`,
  - подтверждение пароля: совпадение с основным полем,
  - имя: `minlength=2`.
- Ошибки Supabase приводятся к человеческому виду: «Invalid login credentials» → «Неверный email или пароль», «User already registered» → «Этот email уже зарегистрирован».
- Поля помечаются классом `.touched` после `blur`, чтобы CSS `:invalid` не подсвечивал форму до первой попытки заполнить.

### 7. Профиль

Отдельная модалка «Мой профиль» открывается из dropdown-меню:
- редактируемое отображаемое имя,
- readonly email,
- плашка текущего тарифа (Free / PRO) — из `profiles.paid`.

Сохранение имени пишет `display_name` в `profiles` через стандартный клиент (RLS разрешает UPDATE собственной строки).

## Auth flow (итоговая схема)

```
[Гость]
  ├─ видит каталог, лендинг, календарь как read-only
  ├─ клик «Забронировать» / «Мои брони» / «Войти»
  │      ▼
  │  [Модалка Auth]
  │      ├─ Вход ────────────┐
  │      └─ Регистрация ─────┤
  │                          ▼
  │                   (валидация на клиенте)
  │                          ▼
  │                   Supabase Auth ──✖️── ошибка → показать
  │                          ▼ ✓
  │                   сессия создана
  │                          ▼
  │                   loadProfile() — создать или прочитать profiles
  │                          ▼
[Пользователь]
  ├─ бронирует (ограничение Free: 3 активных → paywall, hw10)
  ├─ редактирует display_name (модалка «Мой профиль»)
  ├─ выходит через меню аватара
```

## Тестирование

| Сценарий | Ожидание | Результат |
|---|---|---|
| Регистрация с невалидным email | инлайн-ошибка под полем, форма не отправляется | OK |
| Регистрация с паролем 4 символа | инлайн-ошибка «Пароль минимум 6 символов» | OK |
| Регистрация, пароль ≠ подтверждение | «Пароли не совпадают» | OK |
| Вход с неверным паролем | «Неверный email или пароль» | OK |
| После входа — в хедере аватар, не две кнопки | аватар с инициалами имени | OK |
| Меню аватара — email, счётчики, «Профиль», «Выйти» | dropdown закрывается по клику вне | OK |
| Профиль: меняю display_name → в меню обновилось | строка в `profiles` обновлена, UI перечитал | OK |
| Выход — возвращается «Войти», брони скрываются | сессия очищена, RLS отрезал данные | OK |
