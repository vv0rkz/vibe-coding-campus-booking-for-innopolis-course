# Домашнее задание 6 — Подключение базы данных и авторизация

## Проект: CampusBook — сервис бронирования ресурсов кампуса

---

## Что сделано

В репозитории реализована интеграция с **Supabase** (таблица `bookings`, RLS) и **вход по email/паролю**. Каталог ресурсов по-прежнему из константы в `script.js`.

**Режим без ключей:** если в `supabase-config.js` пустые `url` и `anonKey`, сайт работает как раньше — только **localStorage**, без входа и без облака.

**Режим с ключами:** бронирования пишутся в Supabase; без входа форма бронирования скрыта; в шапке — «Войти» / email и «Выйти».

Файлы: `projects/campus-booking/supabase-config.js` (заполнить), `supabase-config.example.js` (шаблон), `supabase/schema.sql` (SQL для Supabase), подключение в `index.html` + логика в `script.js`.

## 1. Настройка Supabase

### Создание проекта
1. Зарегистрировался на [supabase.com](https://supabase.com)
2. Создал новый проект «CampusBook»
3. Получил URL проекта и anon key из Settings → API

### Создание таблицы

Готовый скрипт: `projects/campus-booking/supabase/schema.sql` — выполнить в **Supabase → SQL Editor**.

**Таблица `bookings` (как в коде):**

| Поле | Тип | Описание |
|------|-----|----------|
| id | uuid (PK, default gen_random_uuid()) | ID брони |
| user_id | uuid (FK → auth.users) | Авторизованный пользователь |
| user_name | text | Имя из формы |
| resource_id | text | ID ресурса как в MVP (`r1`, `r2`, …) |
| date | date | Дата |
| start_time, end_time | text | Время `HH:mm` |
| purpose | text | Цель |
| status | text | `active` / `cancelled` |
| created_at | timestamptz | Создано |

Таблица `resources` в БД в этом MVP **не обязательна** — ресурсы берутся из `RESOURCES` в `script.js`.

## 2. Интеграция с фронтендом

В `index.html` подключаются UMD-бандл Supabase, `supabase-config.js` и `script.js`. В конфиге задаётся `window.CAMPUSBOOK_SUPABASE = { url, anonKey }`.

### Данные: localStorage или Supabase

| Операция | Режим без ключей | Режим с Supabase |
|----------|------------------|------------------|
| Бронирования | `localStorage` (`campusbook_bookings`) | таблица `bookings` |
| Ресурсы | константа `RESOURCES` | та же константа |
| Вход | не нужен | Supabase Auth (модальное окно) |

## 3. Авторизация

### Что реализовано

- Форма регистрации (email + пароль) через Supabase Auth
- Форма входа
- Кнопка выхода в header
- Бронирование доступно только авторизованным пользователям
- Гость может просматривать каталог, но при попытке забронировать видит приглашение войти

### Auth flow

```
Открыл CampusBook
       │
       ▼
Видит каталог как гость (без входа)
       │
       ▼
Нажимает «Забронировать»
       │
       ├── Авторизован → переходит к форме бронирования
       │
       └── Не авторизован → показывается форма входа/регистрации
              │
              ▼
       Вводит email + пароль → входит → возвращается к форме
```

### Роли

| Роль | Что может | Как реализовано |
|------|-----------|-----------------|
| Гость | Просмотр каталога, фильтрация, поиск | Без входа, RLS-политика разрешает SELECT на resources |
| Пользователь | Бронирование, просмотр своих броней, отмена | Supabase Auth, RLS-политика на bookings привязана к user_id |
| Админ (план) | CRUD ресурсов, все бронирования | Отдельная роль в Supabase, пока не реализовано |

### Row Level Security (RLS)

В `schema.sql`: **SELECT** — свои строки **или** любые со `status = 'active'` (чтобы на клиенте проверять пересечения по слотам). **INSERT** — только со своим `user_id`. **UPDATE** — только свои строки.

После деплоя на хостинг: в **Supabase → Authentication → URL configuration** добавьте **Site URL** и **Redirect URLs** вашего опубликованного сайта (см. раздел «Деплой» в корневом `README.md`).

## 4. Тестирование

### Что проверено

| Тест | Результат |
|------|-----------|
| Регистрация нового пользователя | Email приходит, аккаунт создаётся |
| Вход существующего пользователя | Сессия сохраняется, имя отображается в header |
| Создание бронирования | Запись появляется в Supabase Table Editor |
| Просмотр своих бронирований | Показываются только свои (RLS работает) |
| Отмена бронирования | Статус меняется на cancelled в БД |
| Выход из аккаунта | Сессия очищается, форма бронирования скрывается |
| Гость пытается забронировать | Видит приглашение войти |
