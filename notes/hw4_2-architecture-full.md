# Архитектура MVP — CampusBook
### Ответы на 16 блоков вопросов по архитектуре

> **Проект:** CampusBook — сервис бронирования ресурсов кампуса (переговорки, коворкинги, оборудование, консультации)
> **Стек MVP:** HTML + CSS + JavaScript (vanilla), localStorage
> **Репозиторий:** https://github.com/vv0rkz/vibe-coding-campus-booking-for-innopolis-course

---

## 1. Продукт и сценарий

**Главная задача MVP:**
Студент может за 30 секунд найти свободный ресурс в кампусе и забронировать его — без переписок в чатах, без конфликтов.

**Пользователь:** студент 18–25 лет, участвует в командных проектах, нужна переговорка/коворкинг/оборудование.

**Ключевой сценарий для защиты:**

```
Студент открывает сайт
        │
        ▼
   Видит каталог ресурсов
        │
        ▼
   Фильтрует по типу «Переговорные»
        │
        ▼
   Выбирает «Переговорная Эврика»
        │
        ▼
   Заполняет форму (имя, дата, 14:00–15:30, «Проектная встреча»)
        │
        ▼
   Нажимает «Забронировать»
        │
        ├── ОК → уведомление + бронь в списке
        │
        └── Конфликт → сообщение об ошибке → исправляет время
        │
        ▼
   Раздел «Мои бронирования» — видит свою бронь
        │
        └── При необходимости — нажимает «Отменить»
```

## 4. Пользовательские действия

```
 #  Действие                           Где происходит
─────────────────────────────────────────────────────────────
 1  Просмотреть каталог                Секция «Доступные ресурсы»
 2  Отфильтровать по типу              Кнопки-фильтры (Все / Переговорные / ...)
 3  Найти по тексту                    Поле поиска — работает в реальном времени
 4  Заполнить форму бронирования       Секция «Забронировать ресурс»
 5  Создать бронь (отправить форму)    Кнопка «Забронировать»
 6  Посмотреть свои брони              Секция «Мои бронирования»
 7  Отменить бронирование              Кнопка «Отменить» → модальное подтверждение
 8  Экспортировать данные              Кнопка «Скачать JSON»
 9  Переключить тёмную тему            Кнопка «🌙 Тёмная» в шапке
10  Войти / зарегистрироваться (hw6)   Кнопка «Войти» в шапке → модальное окно
```

## 5. Результат для пользователя

```
   ДЕЙСТВИЕ             →  ЧТО ВИДИТ ПОЛЬЗОВАТЕЛЬ
──────────────────────────────────────────────────────────────────

   Фильтрация           →  Карточки мгновенно перерисовываются
   по типу                 (без перезагрузки страницы)

   Поиск по тексту      →  Карточки фильтруются при каждом
                           вводимом символе

   Отправка формы       →  ✅ Зелёная плашка «Бронирование
   (успех)                 успешно создано!» на 3 секунды
                           Бронь появляется в списке ниже

   Отправка формы       →  ❌ Красная плашка с деталями:
   (конфликт слотов)       «Ресурс уже забронирован 14:00–15:00»

   Отправка формы       →  ❌ «Время начала должно быть
   (время некорректно)     раньше времени окончания»

   Отмена брони         →  Карточка становится полупрозрачной
                           Статус → «Отменено»
                           Кнопка «Отменить» исчезает

   Экспорт JSON         →  Браузер скачивает файл
                           campusbook_bookings.json
```

## 10. API (черновик для Supabase)

```
  В текущем MVP API нет — всё на клиенте.
  При переходе на Supabase (hw6) используется Auto REST API:

  ┌──────────┬────────┬───────────────────────────┬──────────────────────────┐
  │ Действие │ Метод  │ Endpoint                  │ Ответ                    │
  ├──────────┼────────┼───────────────────────────┼──────────────────────────┤
  │ Все      │ GET    │ /rest/v1/resources        │ [{id, name, type, ...}]  │
  │ ресурсы  │        │                           │                          │
  ├──────────┼────────┼───────────────────────────┼──────────────────────────┤
  │ Мои      │ GET    │ /rest/v1/bookings         │ [{id, resource_id,       │
  │ брони    │        │ ?user_id=eq.{uid}         │   date, status, ...}]    │
  ├──────────┼────────┼───────────────────────────┼──────────────────────────┤
  │ Создать  │ POST   │ /rest/v1/bookings         │ {id, status: "active"}   │
  │ бронь    │        │ body: {resource_id, date, │ или HTTP 409 конфликт    │
  │          │        │  start_time, end_time, …} │                          │
  ├──────────┼────────┼───────────────────────────┼──────────────────────────┤
  │ Отменить │ PATCH  │ /rest/v1/bookings         │ {id, status:"cancelled"} │
  │ бронь    │        │ ?id=eq.{id}               │                          │
  │          │        │ body: {status:"cancelled"}│                          │
  └──────────┴────────┴───────────────────────────┴──────────────────────────┘

  Авторизация: заголовок  Authorization: Bearer <access_token>
  Получение токена:  supabase.auth.signInWithPassword({email, password})
```

## 11. Проверки и правила

```
  ┌────────────────────────────────────────────────────────────────────┐
  │                     ПРАВИЛА ВАЛИДАЦИИ                              │
  ├─────────────────────────┬──────────────────┬───────────────────────┤
  │  Правило                │  Когда           │  Что происходит       │
  ├─────────────────────────┼──────────────────┼───────────────────────┤
  │ Все поля заполнены      │ submit           │ HTML5 required —      │
  │                         │                  │ форма не отправляется │
  ├─────────────────────────┼──────────────────┼───────────────────────┤
  │ startTime < endTime     │ submit / JS      │ ❌ красная плашка      │
  ├─────────────────────────┼──────────────────┼───────────────────────┤
  │ Нет конфликта слотов    │ submit / JS      │ ❌ плашка с деталями  │
  │ (ресурс + дата + время) │                  │ конфликта             │
  ├─────────────────────────┼──────────────────┼───────────────────────┤
  │ Дата не в прошлом       │ input[type=date] │ min = today,          │
  │                         │                  │ прошлые даты серые    │
  ├─────────────────────────┼──────────────────┼───────────────────────┤
  │ Отменять можно только   │ render           │ Кнопка не рендерится  │
  │ активные брони          │                  │ для отменённых        │
  └─────────────────────────┴──────────────────┴───────────────────────┘

  АЛГОРИТМ ПРОВЕРКИ КОНФЛИКТА:
  ─────────────────────────────────────────────────
  Конфликт есть, если ВСЕ условия выполнены:
    1.  b.resourceId === новый resourceId
    2.  b.date       === новая date
    3.  b.status     === 'active'
    4.  НЕ (новый endTime <= b.startTime
             ИЛИ новый startTime >= b.endTime)

  Условие 4 — инвертированная проверка непересечения.
  Если интервалы НЕ расходятся → они пересекаются.
```

## 12. Авторизация

```

  ─────────────────────────────────────────────────────────────
  ПЛАН (hw6 — Supabase Auth):
  ─────────────────────────────

          │
          ▼
  Видит каталог (гость, без входа)
          │
          ▼
  Нажимает «Забронировать» или «Войти»
          │
          ▼
  Модальное окно: [ Вход ] | [ Регистрация ]
          │                │
          ▼                ▼
  email + пароль    email + пароль

          │                │
          └────────┬────────┘
                   ▼
          Supabase выдаёт JWT-токен
                   │
                   ▼
          Сессия сохранена в localStorage
                   │
                   ▼
          Шапка: показывает email + «Выйти»
          Форма бронирования открывается
                   │
                   ▼
          Бронирование → Supabase (с user_id)
                   │
                   ▼
          «Мои бронирования» — только свои (RLS)

  ─────────────────────────────────────────────────────────────
  РОЛИ И ПРАВА (план):
  ─────────────────────────────
  ┌──────────────┬──────────────┬───────────────────────────┐
  │ Роль         │ Вход         │ Права                     │

  ├──────────────┼──────────────┼───────────────────────────┤
```
