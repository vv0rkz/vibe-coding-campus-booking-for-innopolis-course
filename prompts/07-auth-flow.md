# Промпты — hw7: auth flow, профиль, валидация

## Контекст

Есть проект CampusBook (vanilla HTML/CSS/JS + Supabase из hw6). Нужно расширить auth-флоу: таблицу profiles, dropdown-меню с аватаром вместо двух кнопок в хедере, отдельную модалку профиля, нативную валидацию полей.

---

## Промпт 1 — схема profiles

> В моём Supabase-проекте уже есть таблица `bookings` + RLS.
> Добавь таблицу `profiles (id uuid PK → auth.users, email, display_name, paid bool default false, created_at, updated_at)` и политики RLS:
> 1. SELECT — только свой профиль,
> 2. INSERT — только свой профиль, и только с paid = false,
> 3. UPDATE — только свой, но изменять разрешено только display_name (paid обновляется webhook-ом платёжки, не фронтом).
> Плюс триггер auto-updated_at. Ожидаемый формат — один SQL-файл, который можно выполнить в SQL Editor.

## Промпт 2 — хедер и профиль-меню

> В `index.html` сейчас в хедере стоят кнопка «Войти» и рядом с ней при входе ещё и кнопка «Выйти» + email пользователя. Это визуально перегружено.
> Переделай хедер так:
> - одна компактная кнопка «Войти» (SVG-иконка пользователя + подпись) для гостя,
> - аватар с инициалами для залогиненного пользователя, клик по которому открывает dropdown-меню,
> - в меню: имя, email, мини-счётчики (активных броней / всего / тариф), пункт «Мой профиль», кнопка «Выйти».
> Также замени текст-ссылки в навигации на пары «SVG-иконка + подпись» (Каталог / Календарь / Мои брони), и на узких экранах оставляй только иконку.
> SVG — инлайновые symbols в начале body, без внешних CDN.

## Промпт 3 — модалка «Мой профиль»

> Добавь модалку профиля, которая открывается из dropdown-меню.
> В ней: поле «display name» (редактируемое, валидация 2–60 символов), readonly email, плашка тарифа (Free / PRO) из `profiles.paid`.
> По Save — UPDATE profiles set display_name where id = auth.uid(). После успеха — обновить UI (аватар, меню, список «Мои брони»).

## Промпт 4 — нативная валидация

> Переделай формы входа/регистрации и формы профиля:
> - используй HTML5-валидацию (`required`, `type="email"`, `pattern`, `minlength`, `maxlength`),
> - не показывай ошибки пока пользователь не тронул поле (класс `.touched` после blur),
> - для каждого поля — свой `<div class="field-error" data-for="...">`, куда скрипт пишет понятное сообщение,
> - добавь пункт «Повторите пароль» и проверяй совпадение в регистрации,
> - ошибки Supabase приводи к понятным сообщениям на русском: «Invalid login credentials» → «Неверный email или пароль», «User already registered» → «Этот email уже зарегистрирован».

## Промпт 5 — display_name при регистрации

> При регистрации передавай `options: { data: { display_name } }` в `sbClient.auth.signUp`, чтобы имя попало в `user_metadata`.
> В функции `loadProfile()` при первом создании строки `profiles` — бери display_name сначала из `user_metadata.display_name`, потом fallback на локальную часть email.
